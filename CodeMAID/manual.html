CodeCrew Ecosystem

CodeCrew is a local-first AI infrastructure stack. This consolidated environment provides the TUI (Crew), the Dashboard (Maid), and the Engine (MOP). Persistent memory, cascading project context, and self-naming personas — all local, all yours.

Philosophy

Most AI tools start from zero every session. No memory of what you decided last week. No awareness of what project you are working in. No understanding of who you need it to be right now.

CodeMOP solves this with three ideas:

Systematic spine, AI muscles

The routing — which context to load, which persona to activate, which decisions to inject — is all deterministic. Pure logic. Fast, reliable, predictable. The AI does the thinking. CodeMOP does the plumbing.

Context cascades like CSS

Context files live in your project folders and inherit from each other. Root sets the foundation. Each level adds specificity. Most specific always wins. Navigate deeper into a project and your AI automatically knows more about it.

Memory that matters

CodeMOP does not dump entire conversation histories into context. It extracts verified decisions from past sessions and injects only those. Small, precise, no noise. The AI picks up where you left off without being overwhelmed by the journey.

Installation

Step 1 — Start Ollama

ollama serve

Step 2 — Pull a model

# Recommended — good balance of speed and quality
ollama pull qwen2.5:14b

# Smaller fallback
ollama pull qwen2.5:7b

# Tiny — for summarization tasks
ollama pull qwen2.5:4b

Step 3 — Install CodeMOP

cd ~/Projects/ProjectOne
pip install -e . --break-system-packages
Tip The -e flag installs in editable mode. Changes to source files take effect immediately without reinstalling. Perfect for development.

Step 4 — Run the onboarder

codemop-setup

Onboarding

The onboarder runs once on first install. It walks through nine steps and writes everything to disk at the end. Every step can be skipped gracefully if something is not available yet.

Step What happens
1. Welcome Your name is recorded in config
2. Ollama Detects running models, sets preferred and fallback
3. Workspace Sets project root directory and walker depth
4. Personas You select types, Ollama generates personas that name themselves
5. Profile Your default profile assembled from selected personas
6. Connectors OpenWebUI and Pi-Coder auto-detected on common ports
7. Sessions Retention periods set for active and archived sessions
8. Imports Existing sessions from other tools optionally imported
9. Finalize All files written, cron installed, summary shown

To run the onboarder again: 

codemop-setup --reset
Warning Resetting onboarding does not delete your existing personas, profiles, or sessions. It only re-runs the setup wizard. Existing files are safe.

Personas

A persona is an instruction file that defines who your AI is in a given context. Not what it knows — who it is. How it thinks. How it communicates. What it prioritises. What it refuses to do.

Personas are plain text files. YAML frontmatter holds the metadata. Markdown body holds the instruction text the AI receives as its system prompt.

The key thing: personas name themselves. During onboarding, each persona seed is sent to Ollama with a prompt that says introduce yourself and choose a name that fits. Every installation gets a unique team.

Available persona types

Type Style Good for
coder collaborative Architecture, implementation, review
debugger methodical Tracing, isolating, fixing bugs
writer precise Documentation, articles, editing
brainstormer expansive Ideation, architecture, planning
researcher thorough Analysis, fact-checking, synthesis
counselor warm Support, guidance, listening
chef passionate Cooking, baking, nutrition, technique

Profiles

A profile combines personas into a team for a specific context. Where a persona defines who the AI is, a profile defines the composition of the team for a given kind of work.

A profile also sets the preferred and fallback models for that context, so your coding sessions can use a larger model while your quick notes use something faster.

Profiles are referenced by name in rtfm.md files. When CodeMOP assembles context for a project, it loads the referenced profile and resolves all its personas automatically.

Vault Security

The Vault is a multi-tier security model that restricts what the AI can see and do. It is enforced at the API level by the SyntaxGuard, ensuring that safety rules are applied regardless of which UI (CodeMaid or CodeCrew) is used.

Vault Modes

Mode Permissions Restrictions
Safe Read only. Planning. Analysis. No disk writes. No destructive commands.
Cage Read/Write inside project folder. No path escapes (../). No root access.
Free Full system access. Enforces SyntaxGuard (root wipes etc.) only.

Syntax Guard

The Syntax Guard is the active monitoring layer. It scans AI proposals for dangerous patterns before execution.

  • Destructive Patterns: Blocks rm -rf /, mkfs, etc.
  • Hallucination Guard: Detects potential infinite loops in generated code.
  • Path Isolation: Enforces CWD boundaries when in Cage mode.

rtfm.md Files

An rtfm.md file is a context file that lives in a project folder. It tells CodeMOP what profile to use, what the project is about, and any specific instructions for that level of the hierarchy.

The actual rtfm.md file lives in ~/.agents/app/projects/ and is symlinked into your real project folder. CodeMOP manages the symlinks. You never touch them.

Walker finds rtfm.md files by walking up the directory tree from your current working directory. It works exactly like git finding .git/ — start where you are, walk up, stop at the configured root.

The Cascade

Context cascades like CSS specificity. Root sets the foundation. Each level adds detail. Most specific always wins on conflicts. Lists accumulate — tools and personas from every level are combined, not replaced. 

~/Projects/rtfm.md
    model: qwen2.5:7b
    tools: [openwebui]
    personas: [brainstormer]

~/Projects/Programs/rtfm.md
    tools: [picoder]
    personas: [soft-lead-coder]

~/Projects/Programs/CodeMOP/rtfm.md
    model: qwen2.5:14b       ← overrides 7b
    personas: [debugger]

Result assembled by CodeMOP:
    model: qwen2.5:14b       ← most specific won
    tools: [openwebui, picoder]   ← accumulated
    personas: [brainstormer, soft-lead-coder, debugger]
Info The walker respects your configured root boundary. It will never walk above the directory you set as your root in config.yaml. This keeps CodeMOP scoped to your actual work.

Session Memory

CodeMOP does not store raw conversation histories. It extracts the signal from the noise.

When a session file is dropped into the sessions directory, the indexer parses it using the right connector, extracts decisions that were made, and stores them in SQLite. Dead ends — approaches that were tried and abandoned — are also recorded separately so the AI does not repeat them.

When a new session starts, the assembler injects only the verified decisions for that project into the system prompt. The AI knows what has already been decided without being overwhelmed by the full history of how you got there.

Verification

Decisions are extracted automatically but start unverified. You mark sessions as verified in CodeMaid. When a session is verified, all its extracted decisions are marked verified and become eligible for injection into future sessions.

Supported session formats

Format Connector Notes
OpenWebUI HTML export OpenWebUI Fingerprinted automatically
Pi-Coder HTML export Pi-Coder Fingerprinted automatically
JSONL session log Generic CodeMOP's own log format
Plain text / markdown Generic Summarized via Ollama
Unknown HTML Generic Text extracted, summarized via Ollama

Walker

Walker finds rtfm.md files by walking up the directory tree from the current working directory. It checks both the real filesystem path and the mirror path in ~/.agents/app/projects/.

Results are returned root first, most specific last. The assembler then merges them in that order so the most specific values win.

Walker caches its results using an MD5 hash of the rtfm chain. If none of the files have changed, the cached context is served immediately. Any edit to any rtfm.md in the chain invalidates the cache for that path.

Configuration

walker:
  root: ~/Projects    # hard upper boundary
  max_ascent: -1      # how far up from CWD
  max_depth: -1       # how far down from root
  exclude:
    - .git            # never enter these

Assembler

Assembler takes Walker's ordered file list and builds one unified context object. It parses each rtfm.md, merges them using cascade rules, resolves the profile and personas, and builds the final system prompt.

The system prompt has three sections in order: 

1. Persona instructions     ← who you are
   (combined from all resolved personas)

2. Project context          ← where you are
   (accumulated rtfm.md body text)

3. Known decisions          ← what's been decided
   (injected from SQLite by indexer)

Ollama API

The API module bridges CodeMOP and Ollama. It handles health checks, model management, GPU status, chat requests, and session logging. It never crashes — every method has a fallback.

GPU management

On multi-GPU systems Ollama handles the splitting. The API module reads GPU status from Ollama's process API and falls back to nvidia-smi when no model is loaded. Both GPUs on your server are reported separately in CodeMaid's dashboard.

Session logging

Every exchange is logged to a JSONL file in the project's session directory. One file per day. The generic connector can parse these files if they are dropped back into the sessions folder.

Indexer

Indexer watches the sessions directory for new and modified files. When a file appears it: 

1. Hashes the file
2. Checks if already indexed with same hash
3. Picks the right connector
4. Parses the file
5. Writes a JSON sidecar file
6. Registers in SQLite
7. Stores extracted decisions and dead ends

The JSON sidecar means files are only parsed once. Subsequent reads hit the sidecar. The hash check means modified files are automatically re-indexed.

Verification flow

Session file indexed → decisions extracted (unverified)
    ↓
You review in CodeMaid → mark session verified
    ↓
Session and all its decisions marked verified
    ↓
Decisions now eligible for context injection

Cleaner

Cleaner runs on a cron schedule to keep the sessions directory healthy. It applies retention rules to every session file.

Condition Action
Outcome is abandoned Delete immediately regardless of age
Older than active_retention_days, no sidecar Delete — never parsed, no value
Older than active_retention_days, has sidecar Move to archive with sidecar
Older than archive_retention_days in archive Compress with gzip
Decisions table Never touched — decisions are permanent
Tip Run codemop-clean --preview to see exactly what the cleaner would do before it does anything. CodeMaid's preview button uses this same dry-run mode.

Connectors

Connectors parse session files from different tools. Every connector outputs the same standard session object regardless of the source format. The indexer never knows or cares which connector was used.

OpenWebUI connector

Fingerprints HTML by looking for open-webui or openwebui in the first 2KB. Extracts model metadata, finds the overview section OpenWebUI generates, and parses decisions and dead ends from it.

Pi-Coder connector

Fingerprints HTML by looking for pi-coder or picoder. Uses a similar extraction approach to OpenWebUI since both tools generate comparable overview structures.

Generic connector

Handles everything else. Extracts text from HTML, JSONL, markdown, or plain text. Sends the text to the smallest available Ollama model with a summarization prompt. Parses the structured JSON response and stores the results. Always returns something — never fails silently.

Onboarder

The onboarder is a nine-step terminal wizard that runs on first install. It creates all necessary files and configuration from scratch.

The most distinctive step is persona generation. Each selected persona type is sent to Ollama with a seed description and a prompt that asks the model to introduce itself and choose its own name. The model's response is parsed to extract the name and the full introduction text. A second Ollama call then generates the complete instruction set for that persona.

This means no two CodeMOP installations have the same team. Your coder is not someone else's coder. They have their own name, their own voice, their own way of working.

CLI — codemop-setup

# Run onboarder (first time or after reset)
codemop-setup

# Reset and re-run onboarder
codemop-setup --reset

CLI — codemop-index

# Process all existing session files once
codemop-index

# Watch sessions directory for new files
codemop-index --watch

CLI — codemop-clean

# Run cleanup now
codemop-clean --run

# Preview what would happen without doing it
codemop-clean --preview

# Install cron job for automatic cleanup
codemop-clean --install-cron

# Remove cron job
codemop-clean --remove-cron

Guide — Setting Up Your First Project

1. Create the project folder

mkdir -p ~/Projects/Programs/MyProject
cd ~/Projects/Programs/MyProject

2. Create the mirror folder in .agents

mkdir -p ~/.agents/app/projects/Programs/MyProject

3. Create the rtfm.md

cat > ~/.agents/app/projects/Programs/MyProject/rtfm.md << 'EOF'
---
profile: senior-dev
project: MyProject
status: active
git: true
---

# MyProject

What this project is and what you need to know.
EOF

4. Create the symlink

ln -s ~/.agents/app/projects/Programs/MyProject/rtfm.md \
      ~/Projects/Programs/MyProject/rtfm.md

5. Verify walker finds it

cd ~/Projects/Programs/MyProject
python3 -c "
from codemop.walker import Walker
w = Walker()
files = w.walk()
print('Found:', files)
"
Tip CodeMaid will automate all of these steps with a single button click once it is built. For now the manual steps above are the way in.

Guide — Importing Sessions

Drop any session file into the right folder under ~/.agents/app/sessions/. The folder name determines which project the session is associated with. 

# OpenWebUI export for CodeMOP project
cp ~/Downloads/my-session.html \
   ~/.agents/app/sessions/CodeMOP/

# Then index it
codemop-index

Or watch for files automatically: 

codemop-index --watch

After indexing, open CodeMaid to review the extracted decisions and mark the session verified.

Guide — Multi-Machine Setup

The entire ~/.agents/app/ directory can be kept in a private git repository. This syncs your personas, profiles, project contexts, and decisions across machines. 

cd ~/.agents/app

# First machine — initialise repo
git init
git add .
git commit -m "Initial CodeMOP setup"
git remote add origin git@github.com:you/agents-app.git
git push -u origin main
# Second machine — after installing CodeMOP
cd ~/.agents
git clone git@github.com:you/agents-app.git app
Warning Do not commit session files to git. Add a .gitignore to exclude the sessions and archive directories. 
# ~/.agents/app/.gitignore
sessions/
archive/
db/
*.log
*.jsonl

With this setup your personas, profiles, and verified decisions follow you to any machine. Sessions stay local to each machine since they can be large and are already indexed.
